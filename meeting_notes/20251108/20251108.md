## 📃 회의 내용

> 각자 맡은 README 피드백

- 게임 상태 저장 기능 제외.
- Swing GUI 구조 논의
  - `JFrame`은 하나로 통합하고, 각 게임별로 `JPanel`을 교체하는 방식으로 구현.
  - 각 게임에서 자체적으로 View를 구성할지, 아니면 `Swing`에서 View를 만들어 넘겨줄지 논의.
    - 추후 복잡도가 높아질 경우, 각 게임이 자체적으로 View를 구성하는 방식으로 전환.
  - 기본 구조: Application ➡ JFrame 1개 + JPanel 3개 (시작화면 + 각 게임 화면)
  - Swing에서 `User` 객체 사용 X.
  - 실행 흐름: 게임 실행 시 → 시작 화면 표시 → 선택한 게임의 패널 호출.
- 테트리스
  - 키보드 이벤트 필요 여부 논의:
    - 버튼만으로 조작 가능하다고 판단 → 키보드 이벤트는 생략하기로 함.

## 💭 팀 내 고민

- 이벤트 기반 입출력 구조상 DTO 필요성 검토:
  - `Controller`는 버튼 클릭만 인식하고 어떤 그림이 클릭되었는지는 모름 → DTO 필요성 제기.
  - 하지만 `JButton` 자체가 DTO 역할을 할 수 있다고 판단 → DTO 사용하지 않기로 결정.
- Swing 의존성에 대한 논의:
  - Swing에 의존하는 구조가 확장성 측면에서 비효율적일 수 있다는 우려.
  - 그러나 이번 프로젝트의 목적이 GUI 구현이므로 Swing 중심으로 최대한 활용하기로 결정.
